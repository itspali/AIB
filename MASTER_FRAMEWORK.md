# AIB Smart ERP: Core System Architecture & Operational Blueprint

This document serves as the absolute blueprint for the AIB Smart ERP system. All technical planning, API route layouts, and frontend screen mockups discussed or generated by the Cursor Agent must structurally map to this framework.

## 1. Technical Stack Primitives

### 1.1 Frontend Framework & Presentation Core
- **Core Framework**: Next.js (App Router architecture with React Server Components/RSC).
- **Styling Engine**: Tailwind CSS (Leveraging utility-first semantic tokens).
- **Component Primitives**: Radix UI primitives wrapped via Shadcn/UI configurations.
- **State Management & Data Fetching**: React Hook Form coupled with Zod validations for structural inputs; TanStack Query (React Query) for client-side transactional caching and optimistic data updates.

### 1.2 Backend & Infrastructure Engine
- **Database Engine**: Supabase Cloud / Managed PostgreSQL.
- **Database Migrations Handling**: Strict PostgreSQL DDL scripts evaluated sequentially via the native Supabase CLI structure (`supabase/migrations/`).
- **Authorization & Security Context**: Supabase Auth (JWT payload integration with application row-level validation properties).

## 2. Multi-Tenant Operational Methodologies

### 2.1 Absolute Data Isolation Paradigm
- **Database Multi-Tenancy Strategy**: Single database instance utilizing explicit Row-Level Security (RLS) policies. Every operational table contains a `tenant_id UUID NOT NULL` constraint linking back to the master `tenants` identity row.
- **The Security Handshake**: The system prevents database isolation leaks by reading the incoming authenticated session user's App Metadata token context rather than relying on standard frontend variables:
  ```sql
  CREATE POLICY tenant_isolation_policy ON target_table 
      FOR ALL 
      USING (tenant_id = (auth.jwt() -> 'app_metadata' ->> 'tenant_id')::uuid);

2.2 Financial & Calculation Standards
Monetary Storage Fields: Float-type numbers are strictly banned. All wholesale rates, purchase costs, current balances, tax allocations, and calculation vectors must use NUMERIC(15,4) parameters.

Currency Standardization: Multi-currency processing layers must isolate transactions cleanly, consolidating ledger mappings against the parent company's home base_currency code formatted to ISO 4217 alpha-3 uppercase strings (e.g., USD, INR, EUR).

2.3 Temporal & Audit Trail Integrity
Every record throughout the entire schema must preserve timezone-aware historical creation and modification tracking parameters.

Database Format Constraints:

created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL

updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW() NOT NULL

Raw, non-timezone fields (TIMESTAMP WITHOUT TIME ZONE) are strictly prohibited.

3. UI/UX Design System Guidelines
3.1 Three-Zone Application Canvas Layout
All dashboard desktop panels must present a highly unified layout utilizing a standard structural grid container:

Zone A (Top Utility Strip - h-16): Houses global cross-tenant triggers, global search functions (Cmd/Ctrl + K), notification nodes, and tenant profile configuration controls.

Zone B (Left Rail Menu Navigation - w-64): Anchors deep operational groupings (Procurement, Manufacturing, Sales, Financials, Inventory). Supports smooth toggle shrinkage to icon-only presentation modes (w-16).

Zone C (Workspace Container View): Auto-scrolling canvas layer using responsive layout spacing elements (p-4 md:p-6 lg:p-8).

3.2 Mobile Adaptations & Layout Breakpoints
Responsive Navigation Relocation: On compact mobile screens (max-width: 768px), Zone B (Left Menu) hidden flags activate. The system automatically shifts accessible app interactions to a persistent bottom action sheet or drawer component.

Dynamic Grid Form Layout Columns: Form interfaces must prevent layout compression by cascading presentation cells down automatically:

Large Screen viewports (lg): 3 to 4 side-by-side execution layout columns.

Medium Tablet viewports (md): 2 side-by-side presentation columns.

Mobile viewports (sm): 1 full-width stacked cell configuration.

3.3 Progressive Disclosure Input Paradigms
Forms must keep the data entry flow clean to protect onboarding users from cognitive overload.

Form modules must default to an "Essentials Only" view (e.g., Name, Price, SKU). Advanced variables or secondary transaction metadata fields (such as inventory bin locations, custom tracking parameters, or regional tax codes) must be cleanly tucked behind an explicit toggle panel labeled "Show Advanced Fields".
